## Story
**As a** API layer,
**I want** a `ResearchRepository` abstract interface with a `SQLiteResearchRepository` implementation,
**so that** data access is type-safe, testable via mock, and can be swapped to PostgreSQL without touching business logic.

## Acceptance Criteria
1. Abstract `ResearchRepository` class (ABC) in `app/repository.py` with async abstract methods:
   - `create_job(job: ResearchJob) -> None`
   - `update_job_status(job_id: str, status: JobStatus, progress_pct: int, progress_message: str) -> None`
   - `save_evidence(job_id: str, vendor: str, req_id: str, evidence: list[Evidence]) -> None`
   - `save_score(job_id: str, vendor: str, req_id: str, score_result: ScoreResult) -> None`
   - `get_job(job_id: str) -> ResearchJob | None`
   - `get_results(job_id: str) -> ResearchResults | None`
   - `list_jobs(limit: int = 50) -> list[ResearchJob]`
   - `save_final_results(job_id: str, summary: str, rankings: list[VendorRanking]) -> None`
2. `SQLiteResearchRepository(ResearchRepository)` implements all abstract methods using `aiosqlite`.
3. All DML queries use parameterized `?` placeholders (no f-strings in SQL).
4. `get_results()` assembles `ResearchResults` from joined queries: scores + evidence + job summary + rankings_json.
5. `list_jobs()` returns jobs sorted by `created_at DESC`, limited to `limit` rows.
6. FastAPI dependency: `get_repository()` function returns singleton `SQLiteResearchRepository` instance.
7. `MockResearchRepository(ResearchRepository)` in `tests/conftest.py` â€” in-memory implementation for unit tests.

## Dev Notes
**Depends on**: Story 2.1 (models), Story 3.1 (schema).

**Repository singleton pattern**:
```python
# app/repository.py
_repository: SQLiteResearchRepository | None = None

def get_repository() -> SQLiteResearchRepository:
    global _repository
    if _repository is None:
        _repository = SQLiteResearchRepository(DB_PATH)
    return _repository
```

**get_results() assembly logic**:
