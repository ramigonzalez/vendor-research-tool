## Story
**As a** user reviewing completed research results,
**I want** to regenerate the executive summary in a different writing format (Formal, Informal, Concise, Direct),
**so that** I can get a summary that matches my communication needs without re-running the entire research pipeline.

## Acceptance Criteria

1. A new `SummaryFormat` string enum exists in `app/models.py` with values: `formal`, `informal`, `concise`, `direct`
2. Format-specific system prompts exist in `app/prompts/synthesis.py` — each prompt adapts the writing style while preserving the same evaluation data inputs
3. A new abstract method `update_summary(job_id: str, summary: str) -> None` exists on `ResearchRepository` and is implemented in `SQLiteResearchRepository` — it updates ONLY the `summary` column (does NOT touch `completed_at`, `status`, or `rankings_json`)
4. A new `POST /api/research/{job_id}/regenerate-summary` endpoint exists that:
   - Accepts JSON body `{ "format": "formal" | "informal" | "concise" | "direct" }` (defaults to `"formal"` if omitted)
   - Returns 404 if job not found
   - Returns 400 if job status is not `completed`
   - Loads the job's scores, rankings, and requirements from the database
   - Calls the LLM with the format-specific prompt
   - Persists the new summary via `update_summary`
   - Returns `{ "summary": "<new summary text>" }`
5. The endpoint reuses the existing `get_llm()` factory and the same score-highlights builder logic from `generate_summary` in `nodes.py` (extract into a shared helper to avoid duplication)
6. TypeScript type-check passes (`npm run typecheck` in frontend — no frontend changes needed in this story, but types must not break)
7. Python linting passes (`ruff check`)
8. Unit tests cover: (a) each format prompt produces valid LLM input, (b) the `update_summary` repo method, (c) the endpoint returns correct responses for valid/invalid requests

## Dev Notes

### Architecture Context

**Tech Stack** [Source: docs/prd/prd.md#Section 4.1]:
- Backend: Python 3.12, FastAPI, LangGraph, LangChain, aiosqlite
- LLM provider: Configurable via `get_llm()` factory (`app/config.py`) — supports OpenRouter, Anthropic, OpenAI
- Database: SQLite with WAL journal mode

**Existing Summary Generation** [Source: `app/graph/nodes.py` lines 666-711]:
- `generate_summary()` is a LangGraph node that runs at the end of the pipeline
- Builds score highlights: top 3 requirement scores per vendor (max 12 total)
- Uses `SUMMARY_GENERATION_SYSTEM_PROMPT` + `SUMMARY_GENERATION_USER_TEMPLATE` from `app/prompts/synthesis.py`
- Falls back to static string on exception
- Saves via `repo.save_final_results()` which also updates `completed_at` and `status`

**Database Schema** [Source: `app/repository.py`]:
- `jobs` table has `summary TEXT` column — plain text, no format metadata
- `save_final_results()` couples summary save with job completion — the new `update_summary()` method must decouple these

**API Router Pattern** [Source: `app/api/router.py`]:
- Uses FastAPI `APIRouter` with `Depends(get_repository)` for DI
- Existing endpoints: `POST /api/research`, `GET /api/research/{job_id}`, `GET /api/jobs`, `GET /api/research/{job_id}/audit`

**Request Model Pattern** [Source: `app/models.py`]:
- Uses Pydantic v2 `BaseModel` with `field_validator` for domain models
- Enums use `(str, Enum)` pattern for JSON serialization

### Key Implementation Decisions

1. **Shared helper extraction**: The score-highlights builder in `generate_summary()` must be extracted so both the pipeline node and the new endpoint can use it without duplication
2. **Repository decoupling**: `update_summary()` must NOT use `save_final_results()` — that method also overwrites `completed_at` and `status`
3. **Format default**: `formal` matches the current behavior, so existing summaries remain unchanged
4. **No migration needed**: The `summary` column already exists as TEXT — no schema change required
