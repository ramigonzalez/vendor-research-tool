## Story
**As a** user,
**I want** to review the audit trail for past research runs (not just the current one),
**so that** I can go back and examine what queries were generated, which sources were visited, and how scores were computed for any historical job.

## Acceptance Criteria
1. [x] New `audit_events` SQLite table: `(id INTEGER PRIMARY KEY AUTOINCREMENT, job_id TEXT NOT NULL, event_type TEXT NOT NULL, payload_json TEXT NOT NULL, created_at TEXT NOT NULL)`.
2. [x] Index on `job_id` for efficient querying.
3. [x] `save_audit_event()` method in `repository.py` called alongside each `emit_*` in `progress.py`.
4. [x] New endpoint: `GET /api/research/{job_id}/audit` returns events chronologically.
5. [x] Frontend `<AuditView>` fetches audit events when viewing historical jobs via "View Results".
6. [x] ~200 events/run, ~40KB — lightweight storage overhead.

## Dev Notes
**Depends on**: Stories 8.1, 9.4.

**Schema creation** — add to existing `_ensure_tables()` method:
```python
await db.execute("""
    CREATE TABLE IF NOT EXISTS audit_events (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        job_id TEXT NOT NULL,
        event_type TEXT NOT NULL,
        payload_json TEXT NOT NULL,
        created_at TEXT NOT NULL
    )
""")
await db.execute("CREATE INDEX IF NOT EXISTS idx_audit_job_id ON audit_events(job_id)")
```

**Progress.py integration challenge**: The current `emit_*` functions only access the state's progress queue. To persist events, they need either:
1. Access to the Repository instance via state, or
2. A separate persistence callback in state

**Recommended approach**: Add `audit_callback` to state that wraps `save_audit_event()`.

**Storage estimate**: ~200 events/run x ~200 bytes/event = ~40KB/run. Very lightweight.
