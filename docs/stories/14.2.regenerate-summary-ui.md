## Story
**As a** user viewing the executive summary of a completed research job,
**I want** a "Regenerate" button with format options (Formal, Informal, Concise, Direct),
**so that** I can regenerate the summary in a style that fits my audience without re-running the pipeline.

## Acceptance Criteria

1. A "Regenerate" button appears in the `ExecutiveSummary` component header, next to the title — only visible when a summary exists
2. Clicking "Regenerate" reveals a format selector with 4 options: Formal, Informal, Concise, Direct
3. Selecting a format calls `POST /api/research/{job_id}/regenerate-summary` with `{ "format": "<selected>" }`
4. While the API call is in-flight, the button shows a loading spinner and is disabled
5. On success, the displayed summary updates immediately with the new text (no page reload)
6. On error, a brief error message appears near the button (does NOT crash the page)
7. The `SummaryFormat` type is added to `src/lib/types.ts`
8. A `regenerateSummary(jobId, format)` function is added to `src/lib/api.ts`
9. The component needs the `jobId` — it must be threaded through from `ResearchPage`
10. TypeScript type-check passes (`npm run typecheck`)
11. ESLint passes (`npm run lint`)

## Dev Notes

### Architecture Context

**Frontend Tech Stack** [Source: docs/prd/prd.md#Section 4.1]:
- React 19 + TypeScript + Vite
- Tailwind CSS for styling
- No component library — hand-rolled atoms/molecules/organisms pattern

**ExecutiveSummary Component** [Source: `src/components/organisms/ExecutiveSummary.tsx`]:
- Currently receives `results: ResearchResults` as sole prop
- Has `viewMode` toggle (Rendered/Source) — the Regenerate button should sit in the same header area
- Renders `<ReactMarkdown>` for rendered mode, `<pre>` for source mode
- Already guards against null `summary` with a fallback message

**ResearchPage (mount point)** [Source: `src/components/templates/ResearchPage.tsx` line 187]:
- Renders `<ExecutiveSummary results={state.results!} />`
- The `jobId` is tracked in the page's state (from SSE `started` event or URL param) — needs to be identified and threaded through

**API Client** [Source: `src/lib/api.ts`]:
- Uses `apiFetch<T>(path, options)` wrapper over native `fetch`
- Base URL from `VITE_API_URL` env var (defaults to `http://localhost:8000`)
- The new `regenerateSummary` function follows this same pattern

**Existing Button Patterns** [Source: `ExecutiveSummary.tsx`]:
- Toggle buttons use: `px-3 py-1 text-xs rounded-sm border cursor-pointer transition-colors`
- Active state: `bg-accent-primary text-white border-accent-primary`
- Inactive state: `bg-bg-primary text-text-secondary border-border-default hover:border-border-focus`

### Design Decisions

1. **Local state override**: When regeneration succeeds, override the summary locally (don't refetch all results). The `results.summary` prop is the initial value; component manages its own `displaySummary` state
2. **Format selector UX**: A small dropdown/popover (not a modal). Keeps it lightweight and in-context
3. **No confirmation dialog**: Regeneration replaces the summary immediately — the user can always regenerate again with a different format
4. **Error handling**: Inline error text near the button. The ErrorBoundary (Story 13.1) is already in place as a safety net
