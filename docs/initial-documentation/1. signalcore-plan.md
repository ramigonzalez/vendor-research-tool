# SignalCore Vendor Research Tool вҖ” Discovery, Architecture & MVP Plan

## Part 1: Discovery

### 1.1 Business Problem & Value

**Core problem:** An AI engineering team needs to select an LLM observability platform. Comparing vendors is manual, time-consuming, and prone to bias from stale information.

**What SignalCore actually evaluates with this assignment:**
- Can you build an AI-powered research pipeline that goes *beyond* static LLM knowledge?
- Can you design a defensible scoring methodology?
- Can you think architecturally about scale (50 vendors Г— 200 requirements)?
- Do you understand how this maps to their actual product (procurement automation)?

**Success criteria for the prototype:**
1. Evidence is collected from real, recent, citable sources (not just LLM memory)
2. Scoring methodology is transparent вҖ” show the *why*, not just the number
3. Confidence levels reflect evidence quality gaps
4. UI lets a user drill from summary вҶ’ detail
5. Architecture shows thinking about scale even if prototype is minimal

### 1.2 Users & Scale (Prototype vs. Product)

| Vendors | 4 fixed | 50+ dynamic |

**In scope:** Research pipeline + scoring engine + results UI for 4 vendors Г— 6 requirements.

**Out of scope (but discuss on Zoom):**
- Automated vendor software testing (browser/API automation)
- User-defined vendors or requirements (hardcoded for MVP)
- Persistent storage / caching of research results
- Authentication / multi-user
- Real-time updates / streaming UI

### 1.5 Key Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Web search returns low-quality evidence | Medium | High | Use multiple search queries per requirement; validate source recency |
| LLM hallucinates vendor capabilities | High | Critical | Require citations for every claim; confidence scoring penalizes unsourced claims |
| 3 hours isn't enough | High | High | Aggressive MoSCoW; build pipeline first, polish UI last |
| API rate limits during demo | Medium | Medium | Cache results; add fallback to pre-computed data |
| Scoring methodology is too simple | Low | Medium | Document what's intentionally simplified and what scaled version would do |

|---------|----------------|-----------------|

## Part 2: Architecture Design

### 2.1 Solution Strategy

**Architectural approach:** A three-stage pipeline orchestrated by LangGraph.

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Orchestration | LangGraph | Shows graph-based agent thinking; natural fit for multi-step research; aligns with company stack |
| LLM | Claude (Anthropic) | Your primary tool; excellent for structured analysis |
| Web Search | Tavily API | Purpose-built for AI research; LangChain integration; returns structured results with URLs |
| Backend | FastAPI | Async-native; fast to build; Python ecosystem |
| Frontend | Single-page HTML + vanilla JS (or lightweight React) | Minimum viable UI; no build step; renders from JSON API |
| Data | In-memory (Python dicts) | No DB for prototype; document what would be PostgreSQL at scale |

**Key architectural decision: Prevention-first scoring.**
Don't let the LLM generate scores directly. Instead:
1. LLM extracts *evidence* with citations
2. Scoring is *computed* from evidence attributes (source count, recency, directness)
3. LLM provides *narrative justification* constrained by evidence

This mirrors your Validata hallucination prevention philosophy and is exactly what SignalCore would want to see.

### 2.5 Scoring Methodology Design

This is the heart of the assignment. The methodology must be transparent and defensible.

#### Layer 1: Per-Requirement Score (0-10)

For each (vendor, requirement) pair, compute a score based on evidence:

| Factor | Weight | Description |
|--------|--------|-------------|
| **Capability Coverage** | 40% | Does the vendor have this feature? (binary + depth) |
| **Evidence Strength** | 30% | How many independent sources confirm? Quality of sources? |
| **Maturity** | 20% | Is it GA, beta, or experimental? How long available? |
| **Limitations** | 10% | Are there known caveats or restrictions? |

**Scoring rubric:**

| Score | Meaning |
|-------|---------|
| 9-10 | Full native support, well-documented, multiple confirming sources |
| 7-8 | Good support, documented, minor gaps |
| 5-6 | Partial support, workarounds needed, or limited documentation |
| 3-4 | Minimal support, significant gaps or only community-reported |
| 1-2 | Claimed but unverified, or requires significant custom work |
| 0 | No evidence of support |

#### Layer 2: Confidence Score (0-1)

Independent of the capability score. Reflects *how much we trust our assessment*:

| Factor | Weight | Description |
|--------|--------|-------------|
| **Source count** | 30% | More independent sources вҶ’ higher confidence |
| **Source authority** | 30% | Official docs > GitHub > community forums |
| **Source recency** | 25% | Content from last 6 months > older |
| **Consistency** | 15% | Do sources agree or contradict? |

**Why this matters:** A vendor might score 8/10 with 0.9 confidence (strong evidence) or 8/10 with 0.4 confidence (sparse evidence). The user needs to know the difference.

#### Layer 3: Weighted Overall Score

```
weighted_score = ОЈ (requirement_score Г— priority_weight Г— confidence)

Priority weights:
  High   = 3.0
  Medium = 2.0
  Low    = 1.0

Final normalized to 0-100 scale.
```

#### Handling Evidence Gaps (Key Design Question from Assignment)

> "How do you handle requirements where public evidence is strong for one vendor but sparse for another?"

**Approach: Penalize through confidence, not through score.**

- If evidence is sparse, the *confidence* drops, which naturally reduces the weighted contribution
- The score itself reflects our best assessment, but the confidence signals reliability
- In the UI, low-confidence scores are visually flagged (e.g., dashed border, muted color)
- This is more honest than either guessing a score or assigning zero

### 2.6 ADR-001: Evidence Collection Strategy

**Status:** Accepted
**Date:** 2026-02-23

**Context:** The assignment requires going beyond static LLM knowledge. We need to pull fresh evidence from public sources. Options include raw web scraping, search APIs, or structured data extraction.

**Decision:** Use Tavily Search API as the primary evidence collection mechanism, with LLM-generated search queries tailored per vendorГ—requirement pair.

**Rationale:**
- Tavily is designed for AI research use cases, returns structured results with URLs
- Native LangChain integration reduces boilerplate
- Returns snippets we can cite directly
- Alternative (raw scraping) is too slow and fragile for a 3-hour build

**Consequences:**
- (+) Fast to implement, reliable results
- (+) Built-in source URLs for citations
- (-) Depends on Tavily API availability (mitigated: cache results)
- (-) May miss some sources that dedicated scraping would find

### 2.7 ADR-002: Score Computation Strategy

**Status:** Accepted
**Date:** 2026-02-23

**Context:** The LLM could generate scores directly, or we could compute scores from structured evidence attributes.

**Decision:** Hybrid approach вҖ” LLM extracts structured evidence and provides initial assessment, then scores are computed algorithmically from evidence attributes.

**Rationale:**
- Pure LLM scoring is a black box and prone to hallucination
- Pure algorithmic scoring misses nuance that LLM can capture
- Hybrid: LLM does what it's good at (reading comprehension, extraction) while algorithm does what it's good at (consistent, reproducible math)
- This is the same prevention-first philosophy as Ramiro's 7-Layer Reasoning Engine work

**Consequences:**
- (+) Transparent, reproducible scores
- (+) Every score has traceable evidence
- (-) More complex than "just ask the LLM"
- (-) Requires careful prompt engineering for structured extraction

### 2.8 Crosscutting Concerns

| Concern | Prototype Approach | Scale Approach |
|---------|-------------------|----------------|
| Error handling | Try/catch with fallback to cached data | Circuit breakers, retry with backoff |
| Caching | In-memory dict (survives single session) | Redis/PostgreSQL for persistent cache |
| Rate limiting | Sequential API calls | Queue-based with rate limiter |
| Logging | Python logging to stdout | Structured logging (ironically, maybe LangSmith) |
| Configuration | .env file for API keys | Secret manager, feature flags |

### 3.5 Zoom Preparation вҖ” Anticipated Questions & Answers

#### "Why did you structure it as a three-stage pipeline?"

**Answer:** Separation of concerns. Research, scoring, and synthesis have different failure modes. If research fails for one vendorГ—requirement, it shouldn't block scoring of others. At scale, each stage can be independently parallelized вҖ” research is I/O bound (web searches), scoring is CPU-bound (computation), synthesis is LLM-bound. LangGraph makes these stages explicit and testable.

#### "What would change at 50 vendors Г— 200 requirements?"

**Answer:** Three things. First, async task queue (Celery) because 10,000 research tasks can't run synchronously вҖ” you'd need parallel workers. Second, a caching layer (Redis + PostgreSQL) because many requirements will overlap across evaluations and vendor capabilities don't change daily. Third, a smarter research strategy вҖ” instead of NГ—M independent searches, you'd cluster similar requirements and batch research queries. The scoring engine stays the same; it's already deterministic and O(NГ—M).

#### "How do you handle varying evidence quality?"

**Answer:** Through the confidence dimension. Every score has two numbers: the capability assessment and our confidence in that assessment. Low-confidence scores are visually flagged and contribute less to weighted totals. This is more honest than either guessing or assigning zero. The confidence formula weights source authority (official docs > community), count, recency, and consistency. A user can see "Posthog scores 7/10 on evaluation frameworks but we're only 40% confident" versus "Langfuse scores 8/10 and we're 90% confident."

#### "What are the failure modes of your scoring approach?"

**Answer:** Three main ones. (1) *Search quality ceiling* вҖ” Tavily might not find the most relevant pages; mitigated by generating multiple queries per pair. (2) *LLM extraction hallucination* вҖ” the LLM might overstate a capability from a tangential source; mitigated by requiring source URLs and using conservative extraction prompts. (3) *Recency bias* вҖ” newer sources aren't necessarily more accurate; mitigated by weighting authority alongside recency. The biggest structural limitation is that we're assessing *documentation*, not *actual behavior* вҖ” the scale version would include automated testing.

#### "If this were a real product feature, what next?"

**Answer:** Three iterations. (1) *Parameterization* вҖ” let users define vendors and requirements instead of hardcoding. (2) *Evidence persistence* вҖ” cache and index research results so repeated evaluations are instant. (3) *Automated testing* вҖ” this is SignalCore's actual differentiator: don't just read what vendors claim, actually exercise their APIs and measure behavior. The research pipeline becomes one input to a larger evaluation that includes behavioral testing.
